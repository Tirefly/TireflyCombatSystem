# RecalculateAttributeCurrentValues 引入脏标记

---

## Claude

- 问题确认：每次调用都执行完整的 合并 → 排序 → 执行 流程，O(N log N)。
- 评估：这个问题的实际影响取决于使用场景。在大多数游戏中，单个实体的修改器数量通常在 10-30 个量级，全量重算的开销可以忽略。只有在 MOBA 后期或大规模 Buff 堆叠场景下才会成为瓶颈。
- 建议方案：暂不改动。理由：
  - 引入脏标记/增量更新会大幅增加代码复杂度（需要追踪哪些属性被哪些修改器影响、修改器之间的依赖关系）
  - 当前的全量重算保证了正确性，增量方案容易引入一致性 bug
  - 如果未来确实遇到性能瓶颈，可以考虑的方向是：按属性分组缓存合并后的修改器列表，只在相关修改器变化时重算受影响的属性组

## Codex

- RecalculateAttributeCurrentValues 全量合并/排序/执行：
- 可加“修改器缓存 + dirty 标记 + 批量触发（每帧/每批一次）”
- 必要时为 ModifierDef 增加 AffectsAttributes/Deterministic 元数据，允许对确定只影响单属性的修改器做局部重算。

# Pending Removal Request 缺少硬超时自动清理

- 问题确认：TcsStateComponent.cpp:76-111 只有 Warning 日志，没有强制清理。
- 建议方案：
  - 在 UTcsDeveloperSettings 中添加一个可配置的硬超时参数，如 PendingRemovalHardTimeoutSeconds，默认值 30 秒
  - 在 TickComponent 的 Pending Removal 检查逻辑中，当超过硬超时时：
  - 强制停止 StateTree（StopStateTree）
  - 调用 FinalizeStateRemoval 完成清理
  - 输出 Error 级别日志
  - 保留现有的 5 秒 Warning 作为早期预警
- 回复：由于无法准确收集 StateInstance 的 StateTree 在执行过程中创建了多少个 AttributeModifierInstance，所以无法准确释放回收。暂时依旧将该流程完全交给开发者
